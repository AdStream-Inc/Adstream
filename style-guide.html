Adstream CSS Guidelines

Syntax

Use multi line CSS to keep readability high, meaning there should be no single line declarations even if there is only 1 attribute being modified. When dealing with multiple selectors each new selector should be placed on a new line follow by a comma. Always use a trailing semi colon to keep the document less error prone across the lifespan of development.

.example,
.another-example{
		background: blue;
		color: green;
}

When dealing with vendor prefixes keep all colons lined up.

.transition{
		-webkit-transition: all; 
		   -moz-transition: all;  
			    -o-transition: all;
}

Multiple attribute values should be set on a new line and tabbed.

.text-shadow{
		text-shadow:
			 0 1px 0 #000,
			 0 -1px 0 #fff;
}

Selector Formatting

Use hyphen delimited, lowercase selectors e.g. .dropdown-content. Make class names sensible to allow for greater comprehension of what the class is doing in the CSS, .d-cnt is bad .dropdown-content is good. When dealing with prefixes try and giving meaning to the prefix of the selector when applicable. For example if you know that you want to add a selector inside .dropdown then prefix the selector with .dropdown e.g. .dropdown-content. Keep this naming convention when dealing with component states e.g. .is-active or .js-hover.

Selector Specificity

Keep all selectors short, efficient, and portable. Large selectors make your css fragile and do not allow the element to exist in multiple surroundings, for this reason keep selectors less than 3 levels deep. For example .sidebar ul span is not a very efficient selector because the span that is being styled can only live inside of ul that also lives inside of .sidebar. A better solution would be to use .sidebar-title so that the object at hand can live anywhere in the html. Keeping selectors short and explicit makes for greater code reusability and maintainability.

Selectors should not be prefixed with any parent element, this keeps the classes abstract and reusable. For instance lets take the selector ul.nav li a. The ul parent element is not needed here for two reasons; we already know this element being styled is going to be a list, and what if we wanted to use this selector to style both a ul and ol? At its current state that would require us to use two selectors to make sure both ul and ol are styled the same.

Dont Make Assumptions

Instead of building a selector that targets an assumed element such .island a, it is often more robust to put a class name on the element. What happens if we remove that a in our html and replace it with a span? The selector .island a no longer is valid and has to be restructured. Instead use something like .island-tagline that way no matter what element lives inside the .island object it will be styled correctly.

Shorthand

Be extremely explicit when declaring selector attributes. It may be appealing to use background: red, however by using that declaration you are actually saying “I want no image to scroll, aligned top left and repeating X and Y and a background colour of red”. In most cases this will not be an issue but in the seldom occurances it does happen, being explicit can really help out with troubleshooting.

.foo{
		background-color: red;
		margin-left: 4px;
}

Code Reuse and OOCSS

When building components try and keep things as DRY and portable as possible. Instead of building dozens of unique components, try and spot repeated design patterns and abstract them. Start with base "skeleton" components and then add classes onto these to extend their styling for more unique circumstances.

If you have to build a new component split it into structure and skin. Build the structure or "skeleton" of the component using very generic classes that can be reused and then use more specific classes to skin it up and add design treatments.

When developing HTML components try and use as little extra markup as possible. This not only keeps your HTML lean it also increases portability within your CSS and allows your selectors to be even more explicit. As an example when creating a navigation block instead of using:

<div class="nav">
		<ul>
				<li>...</li>
		</ul>
</div>

use

<ul class="nav">
		<li>...</li>
</ul>

Object Extension

Before designing a new component take time to make sure that it has not already been created, or that an existing component can be extended apon to fit your needs. Its worth mentioning the open/close principle in object oriented design. It basically states that components are always open to extension but closed to modification, meaning that with just a single class you can potentially change the structure of a component altogether. Therefore when utilizing CSS objects it is imperative to keep the open/close principle in mind. An easy example of component extension is extending the .nav abstraction for a sites main navigation.

<ul class="nav nav-main">
		<li>...</li>
</ul>

Now all we have to call is our .nav-main selector to modify the "skin" of our navigation, since it is inheriting the structure of the .nav abstraction. As an added bonus if we need to change the structure of the .nav abstraction for any means it can be completely done through the .nav-main class making our code contained and portable.

Layout

Demo of Grid System

All components should be totally free of widths. They should remain fluid and have their widths set by the grid system. Tru not to assign heights to anything but a grid, most components should have their heights adjusted purely from padding or line-height.

The actual layout of the site should exclusively rely on the grid system and nothing else. Think of it as a bookshelf, the grids are the holes to put your "books" or components in. This way we can ensure that everything will degrade gracefully at lower resolutions. Think of the grids as almost static and constant, meaning that no visual or box model styles should be applied to them.

Typography

All typography is automatically sized and set to a consistant vertical rhythm, components should follow this vertical rhythm as well. Line heights should be set without a unit measurement. All font sizes should be applied in em's with a pixel fallback to allow for responsive sizing determined by viewport size.

Before declaring a font size in a selector check and make sure it is not already declared. Header tag sizes have been designed to be interchangeable in size, meaning that if you want a h1 to look like a h3 you can simply use <h1 class="h3"></h1>.

Golden Number

To keep optimal readability always use a "golden number" (or numbers divisible by the golden number) when dealing with margins or paddings. This number is derived from our line height which is determined by taking the base font size (16) and multiplying it by 150% (1.5) which equals 24. Math is awesome.

!important

It is okay to use !important on helper classes that you want to take precedence such as .left{ float: left!important }. Using !important to get yourself out of nasty specificity situations, is not advised. Rework your CSS and try to combat these issues by refactoring your selectors. Keeping your selectors short explicit will help out here massively.
